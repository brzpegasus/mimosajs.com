New Project mimosa new [nameOfProject] [-d/--defaults]

new

The best way to get started with Mimosa is to use it to create a new application/project structure for you. Create a new project like so:

$ mimosa new nameOfApplicationHere

This will kick off a series of prompts that will allow you to pick out the meta-languages, templating library, server, and server templater you'd like to use. When you've finished picking, Mimosa will create a directory at your current location using the name provided as the name of the directory. Inside that directory Mimosa will populate an application skeleton inside an asset directory. The skeleton will include JavaScript, CSS, and micro-template web assets for the selections you made.

If you chose to have a server stack of your own, it will also include the bare essentials for a base Express application, including a simple router and a file that will be used by Mimosa to get Express started. That code will be also be delivered in meta-language of your choice.

Lastly, you will be given a starter group of views for an index page in the view templating language you chose.

If you choose to not have a server of your own, Mimosa will serve your assets for you. Mimosa provides some configuration options for its embedded server to make not having your own server a little less painful than it would be otherwise.

The assets directory has some example code to get you started, and has a group of vendor scripts, like require.js.

The created directory will also contain the configuration file for Mimosa. Almost everything inside of it is commented out, but all the options and explanations for each option are present. The only things that will not be commented out will changes to the configuration for the javascript, css, and template compilers, as well as the server config if you chose something other than the defaults.

Pick the defaults (-d/--defaults)

Should you be happy with the defaults (CoffeeScript, Handlebars, Stylus, Express, Jade) you can bypass the prompts by providing a --defaults flag.

$ mimosa new nameOfApplicationHere --defaults
$ mimosa new nameOfApplicationHere -d

Watch, Compile, Serve mimosa watch [-s/--server] [-c/--clean] [-o/--optimize] [-m/--minify]

watch

Mimosa will watch the configured watch.sourceDir, by default the assets directory. When files are added, updated, or deleted, the configured compilers will execute Mimosa's workflow and keep the watch.compiledDir updated with compiled/copied assets.

To start watching, execute:

$ mimosa watch

When Mimosa starts up, it registers files in your sourceDir all at once. If you have a very large number of files, 1000 images to copy over for instance, Mimosa may open enough files to cause EMFILE issues. This means that Mimosa has opened up too many files at one time, more files than your system allows to be open at once.

To combat this, the watch config has a throttle property. throttle is the number of files Mimosa should handle every 100 milliseconds during the initial Mimosa startup and for subsequent file additions. For instance, if you set the number to 200, and if you have 1000 files, when Mimosa starts up it will process them in 5 chunks of 200, spaced 100 milliseconds apart. It will do the same if, after Mimosa has started, you add 1000 files with a single copy/paste.

You may still run into EMFILE issues after setting throttle, and it might take some playing around with the number to get things just right.

The default for throttle is 0. When throttle is set to 0, throttling is disabled and all files are processed immediately.

Serve Assets (-s/--server)

If you are not already running a server, and you need to serve your assets up, start Mimosa with the server flag.

$ mimosa watch --server
$ mimosa watch -s

By default, this will look for and run a piece of code located at server.path. If you used the Mimosa command line to build your new project, and you chose to have a server of your own, you will have a server.coffee (.iced/.js) at the root of your file structure that will start up an Express app. Mimosa will run the startServer method in this file. You can leave this file as is if you are simply serving up assets, but this gives you the opportunity to build out a much larger app, with numerous routes and views, should that be your goal.

You can change to using an embedded default (non-extendable) Express server by changing the server.useDefaultServer configuration to true. If you chose to not have a server during project creation this setting will have already been changed for you.

Clean Compiled Assets First (-c/--clean)

Use the --clean flag to clean out the compiled directory before watching starts. This has the effect of forcing a recompile and recopy of all of your assets.

Serve Optimized Assets (-o/--optimize)

Start Mimosa watching with the --optimize flag turned on and Mimosa will run the r.js optimizer on start-up and with every javascript file change. So when you point at either the Express or default server's base URL with optimize flagged, you will be served the result of RequireJS's optimization, packaged with Almond.

When --optimize is turned on, Mimosa will also minify your CSS.

To start up with optimization turned on, execute the following:

$ mimosa watch [--server] --optimize
$ mimosa watch [--server] -o

Minify Assets (-m/--minify)

Start Mimosa watching with the --minify flag turned on and Mimosa will use Uglify to compress your JavaScript files as they are compiled and before they are written. --minify has its own setting in the mimosa-config file for choosing which files to exclude from minification.

To start up with minification turned on, execute the following:

$ mimosa watch [--server] --minify
$ mimosa watch [--server] -m

Minify and Optimize Assets (-mo/--minify --optimize)

$ mimosa watch [--server] --minify --optimize
$ mimosa watch [--server] -mo

When combined, the --minify and --optimize options give you more control over the minification strategy of your application. The r.js optimizer by itself is often good enough to handle minifying, compressing and pulling your modules into single files; however, the occasional file does not take kindly to being run through Uglify and will be broken when compressed. The RequireJS optimizer is all or none, you cannot choose to omit a file that isn't being compressed correctly. So if your application doesn't run because of errors in Uglifying, you are forced to inject other steps into your project's life cycle to use both the optimizer and have minified files.

When --minify and --optimize are used together, r.js minification is turned off and the r.js optimizer is just used to pull all of your application's files together into the same file. Mimosa will Uglify your files itself first, before calling the optimizer, and Mimosa gives you the opportunity to not Uglify whatever files are not making it through Uglifying in working order.

Head over to the Optimization tab to learn more about Mimosa optimization options

Clean Compiled Assets mimosa clean [-f/--force]

clean

This command will wipe out any file from the compiledDir that is currently in the sourceDir. Any assets in the compiledDir that are not in the sourceDir are left alone in case they were placed there purposefully. Mimosa makes a pass to clean out the files, and once done, it makes a pass to clean out any directories that are now empty.

Davids-MacBook-Pro:foo dbashford$ mimosa clean
09:45:55 - /Users/dbashford/mim/foo/public has been cleaned.
Davids-MacBook-Pro:foo dbashford$ 

$ mimosa clean

Force Asset Deletion (-f/--force)

Because the clean command will not remove anything in your compiledDir that isn't in your sourceDir, Mimosa may leave behind code that has been renamed. If mimosa watch is running and you delete or rename a file, Mimosa cleans out the compiled version of that file, but if Mimosa is not running and you delete or remove a file, Mimosa has no way of knowing the files left behind are safe for cleaning.

Enter the --force flag. When included, the --force flag will remove the entire compiled directory and all of its contents, no questions asked.

One Time Asset Build mimosa build [-o/--optimize] [-m/--minify] [-r/--removeCombined] [-j/--jade]

build

If you want to compile a set of assets, and you don't want to start up a watching process to do it, you can use Mimosa's build command.

$ mimosa build

This will run through your assets and compile any that need compiling, deliver the results to the public directory, and then exit.

Optimized Build (-o/--optimize)

If you want to build and generate optimized output, provide an --optimize flag. This will compile all the assets, create the r.js optimized files, and minify your compiled CSS.

$ mimosa build --optimize
$ mimosa build -o

Minify Assets (-m/--minify)

If you want to build with minified code, run the build command with the --minify flag turned on. Mimosa will use Uglify to compress your JavaScript files as they are compiled and before they are written. --minify has its own setting in the mimosa-config file for choosing which files to exclude from minification.

Minify and Optimize Assets (-mo/--minify --optimize)

$ mimosa build --minify --optimize
$ mimosa build -mo

Need a bit more control over the minification strategy of your application? Check the section above under watch and the page on optimization to see the benefit of using --minify and --optimize together.

Remove Leftover Files (-r/--removeCombined)

When used in conjunction with --optimize the --removeCombined flag will clean up after the optimization. If, when the optimization is complete, you only want the combined optimized files left behind, this is the flag for you. This flag is a simple pass-through to the r.js optimizer, which takes a removeCombined flag of its own.

The --removeCombined flag must be used with the --optimize flag or Mimosa will error out.

$ mimosa build --optimize --removeCombined
$ mimosa build -or

Compile the Provided Jade Template (-j/--jade)

Should you not be deploying a node/Express app, the --jade flag will compile an .html version of the index.jade that Mimosa provides with its new command. The --jade flag will attempt to compile the index.jade file by feeding the template production level settings. cachebust will be turned off. reload set to false. optimize will be set to true if you also provide the optimize flag.

If the index.jade file is changed to take different variables, or removed or renamed, the jade compilation will fail with warnings explaining the failures. This makes the command somewhat inflexible and if that causes problems for enough folks then some flexibility can be built in. Just file an issue!

$ mimosa build --jade
$ mimosa build -j

Just Watch, Nothing Else mimosa virgin

virgin

Because, after all, a virgin mimosa is just orange juice.

$ mimosa virgin

This command is just the watching, compiling, linting and notifications. No server, no optimizations, no writing the output. Use this if you've already got an application framework, you don't need files served, and you don't need optimizations. But you do want to know if something doesn't compile before you reload the page, and you do want to introduce linting to your workflow.

Install Dependencies mimosa install [-n/--noamd]

install

$ mimosa install [libraryName]

The install command will use Volo to install dependencies for you. Mimosa will first ask where you want to place the dependency, then it will use Volo to go fetch it from GitHub. Whenever possible, Volo tries to get AMD versions of files so you do not have to wrap them. See the Volo documentation for more details.

$ mimosa install jquery

Install Non-AMD dependencies (-n/--noamd)

The only Volo option exposed by Mimosa at this time is the ability to turn on and off whether you are looking for an AMD or non-AMD version. By default Mimosa will set the install to AMD. But if you wish a non-AMD version of a library, provide a --noamd flag.

$ mimosa install backbone --noamd

Update Project Dependencies mimosa update

update

If you installed an application using the new command, you will want to keep your application up to date with the Mimosa skeleton as it evolves and advances.

Use the update command to update the node packages that Mimosa installed inside your application when it was created and to get any new libraries it has included. This command saves you the trouble of needing to keep track of versions of libraries yourself and ensures you'll keep up to date with the versions Mimosa uses.

For now Mimosa will not attempt to update any of the other assets it delivers (routes, example skeleton files, etc) because many of them may have significantly changed from when they were created.

Mimosa will first run npm uninstall for each of the skeleton libraries already in place in your project, and then it will follow up with an npm install on each removed library and each new library that wasn't there originally.

$ mimosa update

Miscellaneous Commands and Options

config

$ mimosa config

If you've already got a project and want to use Mimosa with it, this command will copy the default configuration into whatever directory you are in. You'll likely have some configuration to update to point Mimosa at your source and compiled directories. This is also a good way to get mimosa-config updates when they occur.

Davids-MacBook-Pro:bar dbashford$ mimosa config
10:41:34 - Copied default config file into current directory.
Davids-MacBook-Pro:bar dbashford$

-h/--help

Mimosa includes extensive help documentation on the command line for each of the commands. Give them a peek if you can't remember an option or can't remember what a command does. Use --help or -h to bring up help. 

$ mimosa --help
$ mimosa new --help
$ mimosa [command] --help

Davids-MacBook-Pro:foo dbashford$ mimosa new --help

Usage: new [options] [name]

Options:

-h, --help output usage information
-n, --noserver do not include express in the application setup
-d, --defaults bypass prompts and go with Mimosa defaults (CoffeeScript, SASS, Handlebars)
-D, --debug run in debug mode

The new command will take you through a series of questions regarding what meta-lanauges/compilers you would like in your project. Once you have answered the questions, Mimosa will create a directory using the name provided, and place a project skeleton inside of it. That project skeleton will by default include a basic Express app, with sample routes and views. It will also include some sample assets for the meta-lanauges/compilers you selected.

$ mimosa new [nameOfProject]

If you wish to copy the project skeleton into your current directory instead of into a new one leave off the then leave off name.

$ mimosa new

Pass a 'noserver' flag to not include the basic Express app. With this set up, if you choose to have Mimosa serve up your assets, it will do so with an embedded Mimosa Express app, and not with one inside your project.

$ mimosa new [name] --noserver
$ mimosa new --noserver
$ mimosa new [name] -n
$ mimosa new -n

If you are happy with the defaults (CoffeeScript, SASS, Handlebars), you can bypass the prompts by providing a 'defaults' flag.

$ mimosa new [name] --defaults
$ mimosa new [name] -d

Davids-MacBook-Pro:foo dbashford$

-D/--debug

All Mimosa commands have a debug mode that will print detailed logs to the console for everything Mimosa is doing.

$ mimosa [command] --debug
$ mimosa [command] -D
